# 🧠 Brain Tumor MRI Classification

Multi-class brain tumor classification from MRI scans, comparing a custom CNN trained from scratch against a fine-tuned EfficientNetV2B0 transfer-learning model — with strict train/test isolation, class-imbalance handling, and Grad-CAM explainability.

Built as an applied ML project which calls for reducing premature mortality from non-communicable disease through early detection and treatment. Automated pre-screening tools like this are aimed at supporting — not replacing — radiologists, especially in settings with limited access to specialist review.

---

## Overview

Given an MRI brain scan, the model classifies it into one of four categories:

- **Glioma**
- **Meningioma**
- **Pituitary tumor**
- **No tumor**

Two models are trained and evaluated the same way, on the same held-out test set, so their performance is directly comparable:

1. **Baseline CNN** — a compact convolutional network trained from scratch.
2. **EfficientNetV2B0** — ImageNet-pretrained backbone, fine-tuned in two phases (frozen head → partial unfreeze with BatchNorm layers kept frozen).

## Key Design Decisions

- **Contour cropping** — every scan is cropped with OpenCV to remove the black background/skull margin, so the model learns anatomy rather than empty space.
- **Strict data isolation** — the `Testing/` split is never touched during training, validation, or model selection. It is loaded once and used only for the final evaluation in Section 7, so reported numbers are genuine test performance, not validation performance mislabeled as such.
- **Class-weighted training** to counter any class imbalance, plus `val_loss`-based checkpointing rather than `val_accuracy` — accuracy alone can look fine while the model quietly underperforms on a minority class.
- **BatchNorm frozen during fine-tuning** — when the top layers of EfficientNetV2B0 are unfrozen, BatchNorm layers are explicitly kept in inference mode, since letting them re-estimate running statistics from small fine-tuning batches is a common source of instability.
- **Mixed-precision training** (`mixed_float16`) for faster training on modern GPUs (T4/RTX/P100).
- **Grad-CAM** visualizations show which regions of the scan the model attended to for its prediction, rather than treating the model as a black box.

## Results

Evaluated once on the held-out test set (1,600 images, 400 per class):

| Model | Accuracy | Macro F1 | Weighted F1 |
|---|---|---|---|
| Baseline CNN (from scratch) | **91.56%** | **0.9132** | 0.9132 |
| EfficientNetV2B0 (fine-tuned) | 82.75% | 0.8213 | 0.8213 |

**Notable finding:** the from-scratch CNN outperformed the fine-tuned EfficientNetV2B0 on this dataset. This is a legitimate result, not a training bug — logs show stable convergence with no divergence or NaNs. One possible explanation is the domain gap between ImageNet's natural photographs and grayscale MRI scans, since only the top ~30 layers of the pretrained backbone were fine-tuned; this is consistent with reports elsewhere that backbones pretrained on natural images don't always transfer as cleanly to medical imaging as backbones pretrained on medical images (e.g. RadImageNet). That said, this hasn't been isolated or confirmed here — it's a plausible hypothesis, not a proven cause, and other factors (fine-tuning schedule, how many layers were unfrozen, dataset size) could also be contributing.

Even the stronger baseline CNN has a weak spot worth flagging: **glioma recall is only 0.76**, meaning roughly a quarter of actual glioma cases are missed (misclassified as another class), despite the model's 91.56% overall accuracy. This is exactly the kind of gap that a single accuracy number hides — and in a real screening context, missing gliomas specifically would be the costliest type of error.

Per-class performance and confusion matrices for both models are in the notebook (Section 7).

## Explainability

Grad-CAM heatmaps are generated for one correctly classified sample per class, showing which regions of the scan most *influenced* the model's prediction. This is not the same as tumor localization or segmentation — the highlighted region indicates where the model's attention was concentrated when producing its output, which may or may not precisely align with the tumor's actual boundaries. See Section 8 of the notebook.


> Model files (`.keras`) are not committed to this repository (see `.gitignore`) since they're large binaries generated by running the notebook. Run the notebook to regenerate them, or host them separately (e.g. Git LFS, Hugging Face Hub, or a cloud bucket) if you need to distribute trained weights.

## Dataset

[Brain Tumor MRI Dataset](https://www.kaggle.com/datasets/masoudnickparvar/brain-tumor-mri-dataset) (Nickparvar, Kaggle) — 7,023 MRI images across 4 classes, pre-split into `Training/` and `Testing/` folders.

## Setup

Then open `brain_tumor_combined_pipeline.ipynb` in Jupyter, Kaggle, or Google Colab and run cells in order.

**On Kaggle:** attach the dataset via *Add Data* → search `masoudnickparvar/brain-tumor-mri-dataset`. The notebook auto-detects the path.

**On Colab:** the notebook downloads the dataset automatically via `kagglehub` — no manual API token or `kaggle.json` upload needed. Just run the cell and it handles the download for you.

**GPU strongly recommended** — a full run (both models, both phases) takes roughly 30-45 minutes on a T4.

## Future Work

- Test-Time Augmentation (TTA) for a potential accuracy boost on borderline glioma/meningioma cases.
- Fine-tune a larger portion of the EfficientNetV2B0 backbone (or try a backbone pretrained on medical images) to test whether the domain-gap hypothesis above holds.
- Wrap the best-performing model in a FastAPI `/predict` endpoint returning class, confidence, and a Grad-CAM overlay, with a simple React front-end for interactive demoing.
